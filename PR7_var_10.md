# Вариант 10 - **Кейс: «Прототип системы регистрации HR-заявок»**

---

## 📖 Ситуация

В компании **HRFlow** сотрудники регулярно создают заявки в отдел кадров: запросы на отпуск, больничные, командировки и другие кадровые задачи.

В настоящее время учет заявок ведётся в разрозненных таблицах и файлах, что приводит к ошибкам при согласовании, потере данных и затрудняет контроль выполнения запросов.

С целью повышения эффективности работы руководство компании приняло решение разработать **прототип настольного приложения для регистрации HR-заявок**, а также оценить возможность его дальнейшей автоматизации с использованием технологий **Robotic Process Automation (RPA)**.

В рамках данной практической работы необходимо:

* разработать простое десктоп-приложение для регистрации заявок;
* реализовать базовый сценарий автоматизации взаимодействия с приложением с использованием **Puzzle RPA**.

---

# **Задание 1. Разработка десктоп-приложения**

Необходимо разработать настольное приложение с графическим пользовательским интерфейсом.

Технология разработки выбирается студентом самостоятельно
(например: **Python Tkinter, C# WinForms, Java Swing** и др.).

---

## 🖥️ Требования к интерфейсу приложения

Приложение должно содержать форму регистрации HR-заявки.

### 📝 Поля ввода

* **Employee** — имя сотрудника
* **HR Manager** — имя HR-менеджера

### 🎛 Элемент выбора

* **Request Type** — тип запроса

Возможные значения:

* Vacation
* Sick Leave
* Business Trip

* **Priority** — приоритет заявки

Возможные значения:

* Low
* Medium
* High

---

### 🔘 Кнопки управления

* ➕ **Add** — добавление новой заявки
* 🧹 **Clear Form** — очистка формы

---

## 📊 Таблица заявок

В нижней части окна должна отображаться таблица зарегистрированных заявок со следующей структурой:

| ID | Employee | HR Manager | Request Type | Priority | Status |
| -- | -------- | ---------- | ------------ | -------- | ------ |

При добавлении новой записи статус устанавливается автоматически:

```json
{ "status": "New" }
```

---

## ⚙️ Управление активностью элементов интерфейса

Необходимо реализовать управление доступностью элементов интерфейса:

* кнопка **Add** должна быть **неактивной (`IsEnabled = false`)**, если не выбрано значение в поле **Request Type** (при условии, что все поля обязательны);
* при выборе типа запроса и заполнении всех полей кнопка должна становиться **активной (`IsEnabled = true`)**;
* кнопка **Clear Form** должна оставаться активной всегда.

# **Задание 2. Базовая автоматизация работы с приложением**

После разработки приложения необходимо создать процесс автоматизации в **Puzzle RPA**, имитирующий действия пользователя.

---

## 🤖 Алгоритм работы робота

1. Запустить разработанное приложение.

2. Дождаться появления главного окна.

3. Последовательно обработать все записи из таблицы входных данных.

Для каждой записи робот должен:

* ввести значение в поле **Employee**;
* ввести значение в поле **HR Manager**;
* выбрать значение **Request Type**;
* выбрать значение **Priority**.

4. Проверить состояние кнопки **Add**:

* если кнопка неактивна (`IsEnabled = false`), определить незаполненное поле или невыбранный элемент и заполнить его;
* после активации кнопки (`IsEnabled = true`) нажать её.

5. Повторить действия для всех записей (не менее 5).

6. Проверить появление записей в таблице приложения.

# 🔎 Требования к корректности выполнения

Результат выполнения работы считается корректным, если:

### ✔ Интерфейс приложения

* кнопка **Add** корректно управляется через `IsEnabled`;
* форма не позволяет добавлять пустые записи.

### ✔ Работа робота

* робот корректно обрабатывает все входные данные;
* учитывает состояние элементов интерфейса;
* успешно добавляет записи в таблицу.

### ✔ Данные

* все записи добавлены;
* значения соответствуют входным данным;
* статус каждой записи — `"New"`.
