# Вариант 30 - **Кейс: «Прототип системы регистрации задач техобслуживания»**

---

## 📖 Ситуация

В компании **MaintainFlow** сотрудники регулярно регистрируют задачи технического обслуживания: профилактические работы, ремонтные операции, инспекции и другие сервисные активности.

В настоящее время учет задач ведётся в разрозненных таблицах и файлах, что приводит к ошибкам при планировании работ, потере данных и затрудняет контроль выполнения обслуживания.

С целью повышения эффективности работы руководство компании приняло решение разработать **прототип настольного приложения для регистрации задач техобслуживания**, а также оценить возможность его дальнейшей автоматизации с использованием технологий **Robotic Process Automation (RPA)**.

В рамках данной практической работы необходимо:

* разработать простое десктоп-приложение для регистрации задач;
* реализовать базовый сценарий автоматизации взаимодействия с приложением с использованием **Puzzle RPA**.

---

# **Задание 1. Разработка десктоп-приложения**

Необходимо разработать настольное приложение с графическим пользовательским интерфейсом.

Технология разработки выбирается студентом самостоятельно
(например: **C# WinForms, Java Swing** и др.).

---

## 🖥️ Требования к интерфейсу приложения

Приложение должно содержать форму регистрации задачи техобслуживания.

### 📝 Поля ввода

* **Equipment** — наименование оборудования
* **Technician** — имя техника

### 🎛 Элемент выбора

* **Maintenance Type** — тип обслуживания

Возможные значения:

* Preventive
* Repair
* Inspection

* **Priority** — приоритет задачи

Возможные значения:

* Low
* Medium
* High

---

### 🔘 Кнопки управления

* ➕ **Add** — добавление новой задачи
* 🧹 **Clear Form** — очистка формы

---

## 📊 Таблица задач

В нижней части окна должна отображаться таблица зарегистрированных задач со следующей структурой:

| ID | Equipment | Technician | Maintenance Type | Priority | Status |
| -- | --------- | ---------- | ---------------- | -------- | ------ |

При добавлении новой записи статус устанавливается автоматически:

```json
{ "status": "New" }
```

---

## ⚙️ Управление активностью элементов интерфейса

Необходимо реализовать управление доступностью элементов интерфейса:

* кнопка **Add** должна быть **неактивной (`IsEnabled = false`)**, если не выбрано значение в поле **Maintenance Type** (при условии, что все поля обязательны);
* при выборе типа обслуживания и заполнении всех полей кнопка должна становиться **активной (`IsEnabled = true`)**;
* кнопка **Clear Form** должна оставаться активной всегда.

# **Задание 2. Базовая автоматизация работы с приложением**

После разработки приложения необходимо создать процесс автоматизации в **Puzzle RPA**, имитирующий действия пользователя.

---

## 🤖 Алгоритм работы робота

1. Запустить разработанное приложение.

2. Дождаться появления главного окна.

3. Последовательно обработать все записи из Excel-таблицы входных данных.

Для каждой записи робот должен:

* ввести значение в поле **Equipment**;
* ввести значение в поле **Technician**;
* выбрать значение **Maintenance Type**;
* выбрать значение **Priority**.

4. Проверить состояние кнопки **Add**:

* если кнопка неактивна (`IsEnabled = false`), определить незаполненное поле или невыбранный элемент и заполнить его;
* после активации кнопки (`IsEnabled = true`) нажать её.

5. Повторить действия для всех записей (не менее 5).

6. Проверить появление записей в таблице приложения.

# 🔎 Требования к корректности выполнения

Результат выполнения работы считается корректным, если:

### ✔ Интерфейс приложения

* кнопка **Add** корректно управляется через `IsEnabled`;
* форма не позволяет добавлять пустые записи.

### ✔ Работа робота

* робот корректно обрабатывает все входные данные;
* учитывает состояние элементов интерфейса;
* успешно добавляет записи в таблицу.

### ✔ Данные

* все записи добавлены;
* значения соответствуют входным данным;
* статус каждой записи — `"New"`.
