# Вариант 0

# Кейс: «Прототип системы регистрации сервисных заявок»

## 📖 Ситуация

В компании **TechFlow** сотрудники регулярно создают внутренние сервисные заявки: запросы на техническую поддержку, доступ к корпоративным системам, настройку оборудования и другие рабочие задачи.

Сейчас сотрудники фиксируют такие обращения в разных таблицах и файлах, что приводит к дублированию информации и затрудняет контроль выполнения задач.

Чтобы упростить управление заявками, руководство компании решило создать **простую настольную систему регистрации заявок**, которая позволит сотрудникам быстро добавлять новые обращения и отслеживать их состояние.

Поскольку в будущем компания планирует автоматизировать часть операций с помощью **программных роботов (RPA)**, разработчики решили сразу проверить, насколько удобно будет взаимодействовать с интерфейсом приложения средствами автоматизации.

В рамках данной практической работы необходимо создать **прототип десктоп-приложения для регистрации заявок**, а затем реализовать базовый сценарий автоматизации взаимодействия с этим приложением.

---

# 🧩 Задание 1. Разработка десктоп-приложения

Необходимо разработать простое настольное приложение с графическим пользовательским интерфейсом.

Технология разработки выбирается студентом самостоятельно
(например: **Python Tkinter, C# WinForms, Java Swing** или другие средства разработки).

---

## 🖥️ Требования к интерфейсу приложения

Приложение должно содержать форму регистрации сервисной заявки.

### 📝 Поля ввода

* **Request Title** — тема заявки
* **Employee** — сотрудник, создавший заявку

### 🎛 Элемент выбора

* **Priority** — приоритет заявки

Возможные значения:

* Low
* Medium
* High

---

### 🔘 Кнопки управления

* ➕ **Add Request** — добавление новой заявки
* 🧹 **Clear Form** — очистка формы ввода

---

## 📊 Таблица заявок

В нижней части окна должна находиться таблица зарегистрированных заявок.

| ID | Title | Employee | Priority | Status |
| -- | ----- | -------- | -------- | ------ |

Статус новой заявки устанавливается автоматически:

```json
{ "status": "New" }
```

При нажатии кнопки **Add Request** введённые данные должны добавляться в таблицу.

---

## ⚙️ Управление активностью элементов интерфейса

Для предотвращения создания некорректных записей необходимо реализовать управление доступностью элементов интерфейса.

* Кнопка **Add Request** должна быть **неактивной (`IsEnabled = false`)**, если хотя бы одно обязательное поле формы не заполнено.
* После заполнения всех обязательных полей кнопка должна становиться **активной (`IsEnabled = true`)**.
* Кнопка **Clear Form** должна оставаться активной независимо от состояния формы.

---

# 🤖 Задание 2. Базовая автоматизация работы с приложением

После разработки приложения необходимо создать процесс автоматизации в **Puzzle RPA**, который имитирует действия пользователя при создании заявок.

---

## Алгоритм работы робота

1️⃣ Запустить разработанное приложение.

2️⃣ Дождаться появления главного окна программы.

3️⃣ Последовательно заполнить форму регистрации заявки:

* ввести тему заявки
* указать сотрудника
* выбрать приоритет

4️⃣ Проверить состояние кнопки **Add Request**.

* если кнопка **неактивна (`IsEnabled = false`)**, робот должен определить, какое поле не заполнено, и дополнить данные;
* после того как кнопка станет **активной (`IsEnabled = true`)**, робот должен нажать её.

5️⃣ Добавить в систему **не менее пяти заявок**.

6️⃣ Проверить появление новых записей в таблице.

---

## 🧾 Логирование выполнения процесса

Для отслеживания работы робота необходимо реализовать **журнал выполнения процесса (логирование)**.

Лог может сохраняться:

* в текстовый файл
  или
* в файл формата **JSON**.

В журнале должны фиксироваться:

* запуск приложения
* ввод данных в форму
* добавление заявки
* возможные ошибки
* завершение работы процесса

Пример записи журнала:

```json
{
  "time": "2026-03-15 10:21:04",
  "action": "AddRequest",
  "title": "Access to CRM",
  "result": "success"
}
```

---

## 🧩 Модульность процесса

Для повышения читаемости и повторного использования кода рекомендуется выделять повторяющиеся действия в **отдельные функции или подпроцессы**.

Например:

* функция запуска приложения
* функция заполнения формы заявки
* функция добавления записи
* функция проверки состояния кнопки

Использование функций позволяет сделать сценарий автоматизации более **структурированным и удобным для сопровождения**.
