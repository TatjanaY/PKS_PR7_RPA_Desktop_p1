# Вариант 31 - **Кейс: «Прототип системы регистрации событий»**

---

## 📖 Ситуация

В компании **EventFlow** сотрудники регулярно регистрируют новые события: конференции, вебинары, митапы и другие мероприятия.

В настоящее время учет событий ведётся в разрозненных таблицах и файлах, что приводит к ошибкам при планировании, потере данных и затрудняет контроль подготовки мероприятий.

С целью повышения эффективности работы руководство компании приняло решение разработать **прототип настольного приложения для регистрации событий**, а также оценить возможность его дальнейшей автоматизации с использованием технологий **Robotic Process Automation (RPA)**.

В рамках данной практической работы необходимо:

* разработать простое десктоп-приложение для регистрации событий;
* реализовать базовый сценарий автоматизации взаимодействия с приложением с использованием **Puzzle RPA**.

---

# **Задание 1. Разработка десктоп-приложения**

Необходимо разработать настольное приложение с графическим пользовательским интерфейсом.

Технология разработки выбирается студентом самостоятельно
(например: **Python Tkinter, C# WinForms, Java Swing** и др.).

---

## 🖥️ Требования к интерфейсу приложения

Приложение должно содержать форму регистрации события.

### 📝 Поля ввода

* **Event** — название события
* **Coordinator** — имя координатора

### 🎛 Элемент выбора

* **Event Type** — тип события

Возможные значения:

* Conference
* Webinar
* Meetup

* **Priority** — приоритет события

Возможные значения:

* Low
* Medium
* High

---

### 🔘 Кнопки управления

* ➕ **Add** — добавление нового события
* 🧹 **Clear Form** — очистка формы

---

## 📊 Таблица событий

В нижней части окна должна отображаться таблица зарегистрированных событий со следующей структурой:

| ID | Event | Coordinator | Event Type | Priority | Status |
| -- | ----- | ----------- | ---------- | -------- | ------ |

При добавлении новой записи статус устанавливается автоматически:

```json
{ "status": "New" }
```

---

## ⚙️ Управление активностью элементов интерфейса

Необходимо реализовать управление доступностью элементов интерфейса:

* кнопка **Add** должна быть **неактивной (`IsEnabled = false`)**, если не выбрано значение в поле **Event Type** (при условии, что все поля обязательны);
* при выборе типа события и заполнении всех полей кнопка должна становиться **активной (`IsEnabled = true`)**;
* кнопка **Clear Form** должна оставаться активной всегда.

# **Задание 2. Базовая автоматизация работы с приложением**

После разработки приложения необходимо создать процесс автоматизации в **Puzzle RPA**, имитирующий действия пользователя.

---

## 🤖 Алгоритм работы робота

1. Запустить разработанное приложение.

2. Дождаться появления главного окна.

3. Последовательно обработать все записи из таблицы входных данных.

Для каждой записи робот должен:

* ввести значение в поле **Event**;
* ввести значение в поле **Coordinator**;
* выбрать значение **Event Type**;
* выбрать значение **Priority**.

4. Проверить состояние кнопки **Add**:

* если кнопка неактивна (`IsEnabled = false`), определить незаполненное поле или невыбранный элемент и заполнить его;
* после активации кнопки (`IsEnabled = true`) нажать её.

5. Повторить действия для всех записей (не менее 5).

6. Проверить появление записей в таблице приложения.

# 🔎 Требования к корректности выполнения

Результат выполнения работы считается корректным, если:

### ✔ Интерфейс приложения

* кнопка **Add** корректно управляется через `IsEnabled`;
* форма не позволяет добавлять пустые записи.

### ✔ Работа робота

* робот корректно обрабатывает все входные данные;
* учитывает состояние элементов интерфейса;
* успешно добавляет записи в таблицу.

### ✔ Данные

* все записи добавлены;
* значения соответствуют входным данным;
* статус каждой записи — `"New"`.

---

# ⚠️ Дополнительная проверка устойчивости

Рекомендуется протестировать сценарий с некорректными данными:

| Event | Coordinator | Event Type | Priority |
| ----- | ----------- | ---------- | -------- |
| (пусто) | Test      | Conference | High     |

Ожидаемое поведение:

* кнопка Add остаётся неактивной;
* робот обнаруживает незаполненное поле;
* корректно завершает добавление записи.

# 🧾 Логирование выполнения процесса

Необходимо реализовать логирование работы робота.

Допускается использование:

* текстового файла
* файла формата JSON

В журнале должны фиксироваться:

* запуск приложения
* обработка каждой записи
* добавление событий
* ошибки (при наличии)
* завершение процесса

Пример записи:

```json
{
  "time": "2026-03-15 10:21:04",
  "action": "AddEvent",
  "event": "Tech Conference 2026",
  "result": "success"
}
```
